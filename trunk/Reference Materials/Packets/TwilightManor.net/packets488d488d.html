The client sends this packet when it thinks it's really out of sync, basically as a resync request. The sequence number is ignorable in that case. Obviously if the client spams this packet, it needs kicking.

The move sequence # starts at 0, and is increased by 1 after every step, but if it would become 256, it is instead reset to 1. You should check the move sequence # sent in the 0x02 (move request) packet to see if it is the same as the one that should be next. If so, accept the move, send this packet, and increase move seq #, resetting it to 1 if it passes 255. (This packet is sent in response to 0x02.)
